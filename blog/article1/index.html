Designing corner shapes for Kannada conjuncts

October 9, 2025 • By Arun C K

This is a short process log describing how I approached the corner and knot shapes for stacked conjuncts while designing GTN. It covers research, sketches, digital trials and the OpenType strategies used to implement robust rendering across platforms.

Problem: stacking and visual rhythm

Kannada conjuncts present a challenge when multiple consonants stack or follow with vowel signs. Corner geometry — where two strokes meet or bend — strongly influences optical color and readability in text. My objective was to craft corner shapes that:

  • remain legible at small sizes (12–16px)
  • avoid visual collisions in tight metrics
  • render consistently across rasterizers (browser engines, HarfBuzz, native apps)

Research & sketches

I began by collecting printed samples and screenshots across UI contexts (web, mobile, e-books). Then I made pencil sketches focusing on:

  1. the shoulder of the consonant where ligation occurs
  2. how vowel signs attach without creating dense ink traps
  3. negative space to preserve distinct counters

Digital workflow

In FontLab I translated the sketches into Bézier shapes. Key steps:

  • Start from a few base strokes and reuse them for similar corners to maintain consistency across glyphs.
  • Use boolean operations carefully; clean up nodes to avoid accidental overlaps that hint poorly.
  • Export test builds frequently to verify rendering in the browser and the desktop apps.

Reusable corner components

I created a small library of corner components (left shoulder, right shoulder, rounded knot). These were placed as masters and scaled or mirrored as needed. Reusable components reduce visual drift between related conjunct forms and simplify fixing issues across the family.

OpenType & feature choices

Implementation choices focused on robust behavior rather than visual-only solutions:

  • ccmp for composition/decomposition corrections.
  • locl to enable Kannada-specific alternates in environments that need them.
  • When stacking required special shapes I used rphf/akhn/blwf style lookups and carefully ordered them to avoid unintended substitutions.

Testing & results

Testing is iterative: after each change I generate variable and static builds and load them in:

  • Chrome, Firefox and WebKit-based browsers
  • Android text renderers and select e-readers
  • Desktop apps (Word, LibreOffice) for fallback behavior

The result in GTN is a set of corner shapes that hold up at small sizes and reduce perceived collisions inside dense lines. Using shared components improved fix velocity — a bug in a base corner only needed a single correction that cascaded to derived glyphs.

Conclusion

Solving corner shapes is part drawing, part engineering. A consistent system of components plus careful OpenType lookup ordering made the shapes robust across platforms. If you want to see the glyph set or test the font, visit the GTN page.
